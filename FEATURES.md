# Features & Goals

## Goal

Provide an interactive, browser-based visualization of how LLVM loop transformations change a program's control-flow graph, memory access patterns, and performance characteristics — making compiler internals tangible for students, researchers, and compiler engineers.

## Applicability

- **Compiler courses**: Visualize what loop tiling, unrolling, interchange, and fusion actually do to the CFG.
- **Performance debugging**: Correlate a transformation with cache miss rates, IPC changes, and branch mispredictions.
- **Patch review**: Quickly compare before/after CFGs when reviewing LLVM transform patches.
- **Research**: Generate publication-ready visuals of transformation effects.

---

## What This Tool Covers

**Covered — LLVM middle-end loop transforms (IR-level):**
Loop Tiling, Loop Unrolling, Loop Interchange, Loop Fusion, Loop Distribution, Loop Flatten, Loop Unroll-and-Jam, Loop Rotation, LICM, Loop Vectorization (structural effect), and any Polly transform (result on IR). Anything that changes the CFG between two `.ll` files is visible.

**Covered indirectly — OpenMP/Clang AST loop transforms:**
`#pragma omp tile`, `#pragma omp unroll`, `#pragma omp interchange`, `#pragma omp fuse`, `#pragma omp split` restructure the AST before IR generation. The tool shows their *effect on the emitted IR* (compare IR with vs. without the pragma), but cannot visualize the AST transformation itself.

**Not covered:**
- Clang AST-level visualization (use `clang -ast-dump` for that).
- Backend / MachineIR transforms (software pipelining, hardware loops).
- Polly's internal polyhedral schedules and dependence polyhedra (only the resulting IR).
- Source-line correlation (the tool works at IR basic-block granularity).
- Vector type semantics (vectorized IR is shown but not interpreted as SIMD).

---

## Automated analysis from your IR

For any pair of LLVM IR files you supply, **`./scripts/analyze.sh before.ll after.ll`** runs the project’s automation:

- **`ll-dump`** produces `web/before.json` and `web/after.json` from those IRs (all CFG and analysis fields come from LLVM on that IR).
- **Profiling** writes `web/perf_compare.json` when the right tool exists: **`perf stat`** for CPU IR (averaged runs), **`rocprof`** when the triple looks like a GPU target (`amdgcn`, `nvptx`, `spir`) or when `GPU_ARCH` / `PROFILER=rocprof` is set.
- **Optimization remarks**: set `OPT_PASS` (e.g. `licm`) when running `analyze.sh` so `opt` emits `remarks.yaml` → `remarks.json` for the UI.

The browser only **displays** whatever JSON is present; it does not invent CFG or counter data. **Curated** folders under `web/datasets/` are static snapshots for demos—treat them separately from a fresh `analyze.sh` run.

## Offline vs. AI-Powered Features

### Fully offline (no AI, no internet)

Everything below is generated by `ll-dump` + `run-perf.sh` and rendered client-side:

| Feature | Source |
|---|---|
| CFG nodes, edges, loop regions | `ll-dump` (LLVM `LoopInfo` + `DominatorTree`) |
| Per-block instruction mix (arith / memory / branch / other) | `ll-dump` (opcode classification) |
| Memory access patterns (stride-1, strided, invariant) | `ll-dump` (LLVM `ScalarEvolution`) |
| Loop-carried, flow, anti, output dependencies | `ll-dump` (LLVM `DependenceInfo`) |
| Trip count, induction variable, loop depth | `ll-dump` (LLVM `ScalarEvolution`) |
| Before/after diff, morph animation | Client-side JS (label comparison) |
| Auto-detection of transform type | Client-side JS (heuristic on CFG structure) |
| Performance bar chart (cycles, IPC, cache, branches) | `run-perf.sh` (Linux `perf stat`, Average of 5 runs by default; configurable via `PERF_RUNS`) |
| Per-block cost and impact score | `ll-dump` (weighted cost model × product of trip counts) |
| Critical path per loop | `ll-dump` (longest-cost chain through loop body CFG) |
| Critical path highlighting (nodes + edges) | Client-side JS (toggle via **Show critical path** button) |
| GPU performance chart (VALU, VMEM, SMEM, LDS, L2, waves) | `run-rocprof.sh` (AMD ROCm `rocprof --stats` + hardware counters) |
| GPU-aware rule-based insights (coalescing, occupancy, VALU) | Insight Engine (GPU metric signals + rules) |
| Optimization remarks (Passed/Missed annotations) | `analyze.sh` + `parse-remarks.py` (LLVM `-pass-remarks-output`) |
| **Insight Engine** — 8 continuous signals, 10 composable rules, bottleneck classification, suggestions | `web/insight-engine/` (Phase 1 — deterministic, no AI) |
| Rule-based chat answers (signals, bottleneck, evidence) | Insight Engine + offline LLM explainer |

### AI-enhanced (requires a model provider)

| Feature | What AI adds |
|---|---|
| Richer "✨ Insights" (Phase 2) | The LLM receives the full Phase-1 report (signals, insights, evidence, bottleneck) and produces a natural-language explanation.  It never invents insights — it only formats and contextualises the deterministic output. |
| Conversational "💬 Chat" | Free-form Q&A — ask follow-ups, request alternative transforms, compare with other passes. Context includes the Phase-1 report. |
| Fallback | If AI is unavailable, both features automatically fall back to the deterministic Insight Engine output (Phase 1). |

### How to integrate an AI model (step-by-step)

This web app talks to **OpenAI**, **Anthropic**, **Ollama**, a **custom** OpenAI-compatible URL, or **Cursor Cloud Agents** (via a small local proxy). **Cursor Pro** subscription usage inside the IDE is separate: there is no public HTTPS “chat completions” endpoint you can paste into a random web page without a proxy.

1. **Click ⚙ AI** in the header bar to open the settings dialog.
2. **Choose a provider** from the dropdown:
   - **Anthropic (Claude)** — requires an API key from [console.anthropic.com](https://console.anthropic.com).
   - **OpenAI (GPT-4o / GPT-4)** — requires an API key from [platform.openai.com](https://platform.openai.com/api-keys). (If you use **Bring your own key** in Cursor, it is the same key type—paste it here as OpenAI.)
   - **Cursor Cloud Agents (API key + local proxy)** — uses a [Cursor API key](https://cursor.com/settings) and the official **`https://api.cursor.com`** Cloud Agents flow (`POST /v0/agents`, poll, fetch conversation). Browsers cannot call that API directly (CORS), so run the repo’s proxy:
     ```bash
     python3 scripts/cursor_agent_proxy.py
     ```
     Default: `http://127.0.0.1:8787`. In **⚙ AI**, set **GitHub repository** to a repo your key may target (e.g. `https://github.com/you/llvm-loop-transform-visualiser`), optional **branch**, and the **proxy base URL**. Each **Generate** or **Chat** send starts a **new** Cloud Agent task (async, can take minutes; oriented at repo work—not identical to in-IDE chat).
   - **Local Ollama** — free, no cloud key. Install [Ollama](https://ollama.com), run `ollama pull llama3`, start the server (for browser access you may need `OLLAMA_ORIGINS=* ollama serve` so the page at `http://localhost:8765` can call `http://localhost:11434`).
   - **Custom endpoint** — any OpenAI-compatible API (LM Studio, vLLM, Together AI, etc.): set the base URL to the `/v1` chat endpoint style your server expects.
3. **Enter the model name** (e.g., `claude-sonnet-4-20250514`, `gpt-4o`, `llama3`).
4. **Enter the API key** (OpenAI / Anthropic / Cursor Cloud) or **API endpoint** (Ollama / Custom). For Cursor Cloud, the key is sent only to your **local proxy** (`127.0.0.1`), which forwards to Cursor’s API. Keys are stored in the browser’s **`localStorage`** (not sent to this repo’s servers).
5. **Click Save**.
6. **✨ Insights → Generate** sends the current transformation context (loops, CFG summary, `perf_compare` metrics if loaded) to the model. **💬 Chat** sends the same context plus your messages.

**CORS note:** Calling OpenAI or Anthropic from a **local `file://` page** may be blocked by the browser; serve the UI with **`python3 -m http.server`** (as `analyze.sh` does) or another HTTP server. Ollama must allow your page’s origin (see `OLLAMA_ORIGINS` above).

If the AI call fails (network, CORS, expired key, model down), the tool falls back to the **rule-based** insights and keyword chat.

## Dataset Dropdown (top-left)

The dropdown serves two purposes:

1. **Reference library** — Built-in entries load **static JSON** under `web/datasets/…` (curated for demos/teaching). They are **not** re-run through `analyze.sh` when you open the page; some include illustrative `perf_compare.json` or `transform_insights` written for documentation.

2. **Your analyses** — After `./scripts/analyze.sh before.ll after.ll`, the **authoritative** outputs are `web/before.json`, `web/after.json`, and optionally `web/perf_compare.json` (and `remarks.json` if you used `OPT_PASS`). To bookmark multiple runs in the dropdown, copy those files into `web/datasets/my-run-1/`, etc., and add an `<option value="datasets/my-run-1">…</option>` in `index.html`.

To add your own dataset: create `web/datasets/your-name/` with `before.json`, `after.json`, and optionally `perf_compare.json` / `remarks.json`, then add an `<option value="datasets/your-name">Your Label</option>` to the `<select id="dataset-select">` in `index.html`.

## Feature Reference

### CFG Visualization

| Feature | How to trigger |
|---------|---------------|
| View single CFG | Load `output.json` or a `before.json`; leave **Show diff** off. |
| Side-by-side diff | Click **Show diff**. Nodes are colored: green = new, red = removed, blue = unchanged. |
| Morph animation | Click **Morph** (requires both `before.json` and `after.json`). The before-CFG smoothly animates into the after-CFG. |
| Drag nodes | Click and drag any basic block to rearrange the graph. |
| Zoom / pan | Scroll to zoom, drag background to pan. |
| Hierarchical layout | Click **Hierarchical layout** in the bottom bar for a top-to-bottom Dagre layout. |
| Reset layout | Click **Reset layout** to return nodes to force-directed positions. |
| Dark mode | Click the moon icon (top-right). |

### Node Detail Panel (click any CFG node)

| Feature | What it shows |
|---------|--------------|
| Instruction mix | Stacked color bar — blue (arithmetic), amber (memory), purple (branch), gray (other) — with counts. |
| Memory access pattern | Read/write ratio bar, per-array cards with access type (Sequential/Strided), animated locality cells showing cache-friendliness. |
| Dependencies | Color-coded chips for loop-carried, flow, anti, and output dependencies with distance and description. |
| Loop info | Header, depth, induction variable, trip count. |
| Block cost & impact | Weighted cost (mem×10, branch×2, arith×1) and impact score (cost × Π trip counts). Shown when clicking any node. |
| Critical path | Per-loop longest-cost chain of blocks. Shown in loop detail; amber-highlighted nodes/edges when **Show critical path** is active. |
| Optimization remarks | If `remarks.json` is loaded (via `OPT_PASS=licm ./scripts/analyze.sh …`), Passed/Missed/Analysis remarks appear per function in the detail panel. |

### Critical Path Visualization

| Feature | How to trigger |
|---------|---------------|
| Toggle critical path | Click **Show critical path** in the bottom bar. Amber border with glow on critical-path nodes, amber edges connecting them, and a **small numeric tag** (scaled impact score, e.g. `13.3k`) above the node. The legend notes that the number is the impact score (cost × Π trip counts). |
| Cost & impact in detail | Click any node — "Block cost" and "Impact score" rows appear. Impact = cost × product of enclosing loop trip counts. |
| Critical path in loop detail | Click a loop node — "Critical path" row shows the ordered block sequence (e.g., `inner → body → inner_step`). |

### GPU Profiling (requires `rocprof`)

| Feature | How to trigger |
|---------|---------------|
| GPU benchmark chart | Run `GPU_ARCH=gfx90a ./scripts/analyze.sh before.ll after.ll` (or set `PROFILER=rocprof`). Auto-detected if IR has GPU target triple. Shows VALU, SALU, VMEM, SMEM, LDS, waves, L2 hit rate. |
| GPU-aware insights | The ✨ Insights panel and rule engine produce GPU-specific analysis: coalescing, occupancy, VALU utilization, LDS usage, L1→L2 traffic. |

### Benchmark Comparison (requires `perf_compare.json`)

| Feature | How to trigger |
|---------|---------------|
| Grouped bar chart | Loads automatically with a **CPU** or **GPU** tag in the heading when `profiler` is `rocprof` or GPU-shaped metrics are present. Shows normalized before/after for execution time, cycles, IPC, L1/LLC miss rates, branch miss rate, instructions (CPU) or kernel-oriented GPU metrics (VALU, VMEM, etc.). Bars animate on load in GPU mode. **"Average of N runs"** appears once in the heading when `"runs"` is set. |
| Absolute values toggle | Click **Show absolute values** above the chart. |
| Speedup summary | Shown automatically — **absolute ratio** (e.g., `1.25x`), not a percent; ≥ 1.0 means after is faster on the chosen time/cycles metric. |
| Transformation insights | Scrollable panel below the chart with "What improved", "What regressed", and "Trade-offs" sections. |

### Dataset Switching

| Feature | How to trigger |
|---------|---------------|
| Switch transformation | Use the dropdown at top-left. Ships with curated examples (tiling, unrolling, interchange, fusion, LICM, IR-derived samples, optional GPU demo) plus any folders you add under `web/datasets/`. |
| Add your own dataset | Create `web/datasets/your-transform/` with `before.json`, `after.json`, and optionally `perf_compare.json`. Add an `<option value="datasets/your-transform">` in `index.html`. |

### Auto-Detection

| Feature | How to trigger |
|---------|---------------|
| Transform type detection | Load `before.json` and `after.json` without a `transform` field. The tool infers tiling, unrolling, interchange, fusion, distribution, LICM, or rotation from structural differences (loop counts, depths, induction variables, memory patterns, per-block cost changes). Shown as "Loop Tiling (detected)" in the badge. When the before/after are structurally identical with no detectable difference, no badge is shown (avoids false positives). |

### Insight Engine (Deterministic — no AI needed)

The tool includes a modular **Insight Engine** (`web/insight-engine/`) documented in [ENGINE.md](ENGINE.md).

| Feature | Description |
|---------|-------------|
| 8 continuous signals | `locality`, `cache_efficiency`, `parallelism`, `vectorization`, `memory_pressure`, `branch_overhead`, `loop_complexity`, `critical_path_dominance` — all in [0, 1]. |
| 10 composable rules | Each rule checks signal deltas and outputs structured insights with confidence scores and evidence.  No hardcoded transformation labels. |
| Bottleneck classification | Argmax over weighted anti-signals: `memory-bound`, `compute-bound`, `control-flow-bound`, `loop-complexity-bound`, or `balanced`. |
| Suggestions | Deterministic, signal-driven suggestions derived from before/after deltas. |
| Phase 2 (LLM) | When an AI provider is configured, the Phase-1 report is sent to the LLM for natural-language explanation. The LLM never overrides the rule engine. |
| Phase 3 (ML — skeleton) | Data-driven bottleneck prediction via a to-be-trained model. Currently produces a placeholder output. |

### AI-Powered Insights

| Feature | How to trigger |
|---------|---------------|
| Configure AI provider | Click **⚙ AI** in the header. Choose OpenAI, Anthropic, local Ollama, or a custom endpoint. Save your API key. |
| Generate insights | Click **✨ Insights** → **Generate**.  Phase 1 (deterministic signals + rules) always runs first.  If an AI provider is configured, the Phase-1 report is sent to the LLM (Phase 2) for explanation — with the full deterministic analysis collapsible underneath. Without AI, Phase 1 is shown directly. |
| AI chatbot | Click **💬 Chat**. The panel uses the Phase-1 engine report for offline answers (signals, bottleneck, evidence) or sends it as context to the configured LLM for richer conversation. |

### Full Pipeline (CLI)

| Feature | How to trigger |
|---------|---------------|
| **C source → visualization (recommended)** | `./run_pipeline.sh input.c loop-unroll`. Auto-detects LLVM tools, compiles C→IR, applies the pass, runs `ll-dump` + profiler + remarks, and serves the UI. Flags: `--port N`, `--no-serve`. |
| From pre-existing IR | `./scripts/analyze.sh before.ll after.ll`. Builds `ll-dump` if needed, writes `web/before.json` / `web/after.json`, runs **`perf stat`** or **`rocprof`** (auto or `PROFILER=`), optional remarks if `OPT_PASS` is set, then serves `web/`. |
| Perf-only (CPU) | `./scripts/run-perf.sh before.ll after.ll web/` → `perf_compare.json`. Default 5 runs; `PERF_RUNS=N …`. |
| GPU profiling | `./scripts/run-rocprof.sh before.ll after.ll web/` or `GPU_ARCH=gfx90a ./scripts/analyze.sh …` when IR targets GPU. Requires ROCm `rocprof`. |
